# tico 引擎演进计划

这个文档用于跟踪 `tico` 朝“可复用、好用的 TUI 2D 游戏引擎”演进的改造路径。

## 目标

- 让核心运行时能力可以被多种游戏类型复用。
- 减少业务代码直接依赖原始 `setTimeout()`、`Date.now()` 或场景内零散计时器。
- 推进内容层的数据驱动，但不强迫把所有行为都塞进 JSON。
- 保持引擎体量小、结构清晰，并且适合终端游戏开发。

## 设计原则

- 运行时服务要对场景安全，且容易清理。
- 引擎 API 优先使用简单对象和显式 owner，而不是隐式魔法。
- 数据负责描述内容，代码负责实现复杂行为。
- 示例项目应该展示引擎能力，而不是绕开引擎能力。

## 阶段规划

### 阶段 1：运行时时间服务

状态：本次迭代已完成

已交付：

- 新增 `EngineTime` 作为公开运行时服务。
- 提供 `after()`、`every()`、`nextFrame()`、`cancel()`、`cancelByOwner()`。
- 将时间服务接入 `GameEngine` 和 `EngineApp`。
- 在场景解绑时自动清理归属于该场景的时间任务。
- 更新 Tetris 示例，演示调度器驱动的 UI 行为。
- 补充引擎测试并同步更新文档。

验收标准：

- 新游戏可以通过 `app.time` 编排延时和循环玩法逻辑。
- 场景切换后不会遗留属于旧场景的时间任务。
- 调度行为与引擎暂停、倍速语义保持一致。

### 阶段 2：数据驱动内容层

状态：下一阶段

范围：

- 规范资源命名空间和内容包约定。
- 为 JSON 内容增加轻量 schema 校验辅助。
- 继续把更多示例内容外置为 JSON 或声明式表：
  - 波次定义
  - 敌人数值
  - 掉落表
  - UI 文案
- 保留“模式 id + 代码处理器”的复杂行为实现方式。

验收标准：

- 示例游戏可以在不改多处业务逻辑的前提下调整内容和平衡性。
- 资源加载与缓存清理规则保持可预测。

### 阶段 3：玩法组合能力

状态：规划中

范围：

- 明确 scene / system / gameplay module 的生命周期模式。
- 为同时需要 `Entity` 与 `Node2D` 能力的对象补一层轻量桥接模式。
- 增加覆盖层、转场、调试 HUD 等轻量运行时助手。
- 把 owner 级清理能力扩展到时间调度之外的更多服务。

验收标准：

- 开发一个新游戏场景时，主要工作是组合引擎能力，而不是重复搭基础设施。

### 阶段 4：开发体验

状态：规划中

范围：

- 扩展文档中的 recipes 和架构范例。
- 为常见终端游戏类型提供 starter template。
- 加强公共 API 与示例用法的测试覆盖。

验收标准：

- 新贡献者只看文档和示例，就能快速做出一个可玩的原型。

## 当前建议

下一步建议优先推进阶段 2，并控制范围：先做内容 schema、资源约定，再挑一个示例继续往数据驱动升级。
